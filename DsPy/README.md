# DSPy Cybersecurity SOC Assistant

A production-style demonstration of [DSPy](https://github.com/stanfordnlp/dspy) applied to Security Operations Center (SOC) workflows. Uses only free, open-source datasets and local LLMs via Ollama — no API keys, no data leaving your machine.

## What this project is

Traditional LLM-based security tools are built by hand-crafting prompts. When the model changes, the prompts break. When accuracy is poor, you iterate on strings. This project takes a different approach using **DSPy** — a framework where you declare *what* you want the LLM to do (Signatures), compose those declarations into pipelines (Modules), and let the optimizer automatically improve them using your own labelled data.

The result is a three-pipeline SOC assistant:

| Pipeline | Task | DSPy concept demonstrated |
|---|---|---|
| **Threat Classifier** | Classify network logs into threat categories | `Signature`, `Predict`, `ChainOfThought` |
| **CVE Analyst** | Assess CVE impact on a given infrastructure | `MultiChainComparison`, `Typed outputs` |
| **Alert Triage** | Prioritize a queue of SOC alerts | `Module` composition, `Assert` constraints |

## DSPy concepts covered

```
dspy.Signature          — typed contract declaring LM inputs/outputs
dspy.InputField         — annotated input variable with description
dspy.OutputField        — annotated output variable the LM must produce
dspy.Predict            — basic single-call predictor
dspy.ChainOfThought     — adds auto reasoning field before outputs
dspy.Module             — composable pipeline unit (like nn.Module)
dspy.Assert             — runtime constraint on LM outputs
dspy.Example            — labelled data point for eval and optimization
dspy.Evaluate           — runs metric function over a test set
dspy.BootstrapFewShot   — optimizer: auto-selects few-shot examples
dspy.MIPROv2            — optimizer: rewrites instructions + examples
dspy.inspect_history    — debug: shows compiled prompts sent to LM
```

## Datasets used (all free/open-source)

| Dataset | Source | Used for |
|---|---|---|
| CICIDS 2017 | Canadian Inst. for Cybersecurity | Network intrusion classification |
| NVD CVE feed | NIST (public API) | CVE impact analysis |
| UNSW-NB15 | UNSW Canberra | Supplementary network anomaly data |
| Synthetic SOC alerts | Generated by `data/generate_alerts.py` | Alert triage training/eval |

All datasets have fallbacks — if offline, synthetic data is generated locally.

## Quickstart

### 1. Install Ollama and pull a model

```bash
# Download from https://ollama.ai, then:
ollama serve                        # start the server
ollama pull llama3.2                # recommended (2GB, strong reasoning)
# or for lower RAM:
ollama pull phi3.5                  # 2.2GB, excellent instruction following
ollama pull qwen2.5                 # 0.4GB, fastest on CPU
```

### 2. Install Python dependencies

```bash
python -m venv venv
venv\Scripts\activate               # Windows
# source venv/bin/activate          # Mac/Linux

pip install -r requirements.txt
```

### 3. Download datasets

```bash
python data/loader.py               # downloads + formats all datasets
                                    # falls back to synthetic if offline
```

### 4. Run notebooks in order

```bash
jupyter notebook notebooks/01_dspy_concepts.ipynb
```

## What each notebook teaches

**01_dspy_concepts.ipynb** — pure DSPy fundamentals with no domain knowledge required. You see a Signature compiled into a prompt live, compare `Predict` vs `ChainOfThought` outputs side by side, and inspect what DSPy actually sends to the LLM.

**02_threat_classifier.ipynb** — classifies network log entries (protocol, bytes, duration, flags) into threat categories (DDoS, Port Scan, Brute Force, Normal). Uses real CICIDS 2017 data. Demonstrates how CoT reasoning improves accuracy on ambiguous samples.

**03_cve_analyst.ipynb** — takes a CVE ID or description, cross-references infrastructure context, and produces an impact assessment with remediation steps. Uses live NVD API data. Demonstrates typed output fields and multi-step module composition.

**04_alert_triage.ipynb** — takes a queue of SOC alerts and produces a priority-ranked list with analyst notes. Demonstrates DSPy `Assert` constraints (output must be valid JSON, severity must be HIGH/MEDIUM/LOW), and shows `BootstrapFewShot` optimization improving priority accuracy.

## Key insight: why DSPy over prompt engineering?

```python
# Prompt engineering approach — brittle, model-specific:
prompt = f"""You are a SOC analyst. Classify this log as DDoS, PortScan, or Normal.
Log: {log_entry}
Think step by step. Output format: CATEGORY|CONFIDENCE|REASON"""

# DSPy approach — declarative, optimizer-friendly:
class ThreatClassifier(dspy.Signature):
    """Classify network log entries into threat categories."""
    log_entry: str     = dspy.InputField(desc="Raw network log line")
    threat_type: str   = dspy.OutputField(desc="One of: DDoS, PortScan, BruteForce, Normal")
    confidence: float  = dspy.OutputField(desc="Confidence score 0.0-1.0")
    reasoning: str     = dspy.OutputField(desc="Step-by-step classification rationale")
```

With DSPy: swap models without rewriting prompts, run an optimizer to improve accuracy automatically, save and load the optimized program.
